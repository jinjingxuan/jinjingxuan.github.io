git的回退命令

首先将本地的内容回退到自己想要的版本。回退版本我一般使用

git reset

其中这条命令中又分为两种

git reset –hard ：会改变本地源码

git reset –soft ：改变head的指向，本地代码不会变

回退到某个版本可以使用：

git reset –soft xx版本号xxxx

git reset –hard xx版本号xxx

如果只是回到上个版本，可以使用

git reset –soft HEAD~1

git reset –hard HEAD~1

场景：我提交了push之后发现push的代码存在错误，应该怎么操作？

  • git reset –soft HEAD~1 ：将head指向上一个版本
  • git status：会发现已经提交的内容会再次出现在缓存中
  • git add . ：重新add
  • git commit -m ‘’ : 重新填写信息
  • git push：会报错
  • git push –force ：当前分支的版本低于远程分支的版本，所以要想覆盖掉它，必须使用force

请求头中的contentType

  • 当添加contentType："application/json"时，一般用于处理复杂的json对象，F12检查时

  • 当不添加 contentType："application/json"的时候可以向后台发送json对象形式，F12检查时

  • 当添加contentType: 'application/x-www-form-urlencoded'时

请求头字段详解

  • Cache-Control:no-cache：告诉（代理）服务器不直接使用缓存，直接向原服务器发起请求
  • Pragma:no-cache：不缓存页面
  • Pragma是HTTP / 1.0实现，并且cache-control是相同概念的HTTP / 1.1实现
  • Connection:keep-alive：网络连接就是持久的，不会关闭，使得对同一个服务器的请求可以继续在该连接上完成。
  • Content-length:10：用来说明传输的正文大小
  • Host：描述请求将被发送的目的地，包括，且仅仅包括域名和端口号。
  • Origin：用来说明请求从哪里发起的，包括，且仅仅包括协议和域名。 这个参数一般只存在于CORS跨域请求中，可以看到response有对应的header：Access-Control-Allow-Origin
  • Referer：告知服务器请求的原始资源的URI，其用于所有类型的请求，并且包括：协议+域名+查询参数
  • User-Agent会告诉网站服务器，访问者是通过什么工具来请求的

响应头字段详解

  • Access-Control-Allow-Credentials:true：响应头是否展示在浏览器中
  • Access-Control-Allow-Origin：跨域
  • Server:Tengine：服务器
  • Transfer-Encoding:chunked：分块编码
